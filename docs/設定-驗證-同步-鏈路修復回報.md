# 設定儲存 → 驗證狀態 → 同步資料 → Dashboard 讀取 — 鏈路修復回報

## 1. 問題根因

### 1.1 驗證狀態未持久化（Bug 1）

- **現象**：設定頁「測試連線」成功後當下顯示成功，儲存、離開再回來後又全部變成「尚未驗證」。
- **根因**：驗證結果僅存在前端 `ApiConnectionSection` 的 `useState`，從未寫入後端；重新載入或 refetch 後元件以初始 `idle` 渲染。
- **後端**：`UserSettings` 與 `settingsSchema` 未包含驗證相關欄位；`saveSettings` 只持久化表單欄位；`test-connection` 只回傳 JSON，未呼叫 storage 寫回。

### 1.2 儲存後畫面未反映（Bug 2）

- **現象**：儲存成功後，驗證燈號／文案未更新為上次結果。
- **根因**：儲存後雖有 `invalidateQueries(["/api/settings"])`，但連線區塊的 state 只以「初始 props」建構，未依 refetch 後的 `settings` 同步；且未區分「欄位值是否變更」以決定是否保留驗證。

### 1.3 驗證成功 ≠ 資料已同步（流程缺口）

- **現象**：Facebook 驗證成功後，dashboard / 今日決策中心仍多處為 0。
- **根因**：
  - 儲存設定**不會**自動觸發 Facebook / GA4 帳號同步；同步需使用者手動呼叫 `POST /api/accounts/sync`。
  - Dashboard 讀取的是 **batch cache**（`getLatestBatch`）；無同步則無帳號，無擷取則無 batch，因此數字為 0 或空。
  - 空狀態時 UI 未區分「尚未同步」「已同步但尚未擷取」「無資料」，易被解讀成「廣告真的都是 0」。

## 3. 驗證狀態目前如何持久化

- **寫入時機**：`POST /api/settings/test-connection` 在每次成功或失敗回傳前呼叫 `storage.patchVerificationStatus(userId, type, payload, value)`。
- **寫入內容**（依 type 寫入對應欄位）：
  - `fbStatus` / `gaStatus` / `aiStatus`：`"success"` 或 `"error"`
  - `fbVerifiedAt` / `gaVerifiedAt` / `aiVerifiedAt`：`checkedAt`（ISO 字串）
  - `fbLastError` / `gaLastError` / `aiLastError`：失敗時為後端回傳的 `message`，成功為 `null`
  - `fbValidatedValueHash` / `gaValidatedValueHash` / `aiValidatedValueHash`：僅在成功時寫入 `valueFingerprint(value)`（length + 前 2 字 + 後 2 字，不存密文）
- **失效時機**：`PUT /api/settings`（儲存設定）時，若某欄位新值的 fingerprint 與既有 `*ValidatedValueHash` 不同，則將該類型之 status 重設為 `idle`、清空對應 VerifiedAt / LastError / ValidatedValueHash。
- **讀取**：`GET /api/settings` 回傳完整 `UserSettings`（含上述欄位）；設定頁以 `buildInitialResultFromSettings(type, settings)` 轉成 `ConnectionResult` 作為 `initialResult`，並用 `useEffect` 在 settings 更新後同步到燈號與文案。

## 4. Facebook / GA4 同步流程怎麼補

- **不自動觸發**：儲存設定或測試連線成功後，**不會**自動呼叫 `POST /api/accounts/sync`，以避免未預期的長時請求與錯誤。
- **設定頁**：
  - 在 API 綁定卡內新增「資料同步狀態」區塊，顯示：
    - Facebook：已同步 N 個廣告帳號、最後同步時間
    - GA4：已同步 N 個 Property、最後同步時間
  - 按鈕「立即同步帳號」：呼叫 `POST /api/accounts/sync`，完成後 invalidate `/api/accounts/synced` 與 `/api/settings`，並 toast 結果。
- **今日決策中心**：仍使用既有「更新資料」流程（可先選帳號與區間再觸發 refresh）；refresh 邏輯內會依 selected 帳號自動補齊尚未在 synced 列表的 Meta 帳號再擷取。

## 5. Dashboard 如何區分「0 數據」與「未同步」

- **API**：`GET /api/dashboard/cross-account-summary` 在無 `batch.summary` 時改為：
  - 依 `storage.getSyncedAccounts(userId)` 計算是否有 Meta/GA4 同步帳號；
  - 回傳 `dataStatus`：`no_sync`（尚未同步）、`synced_no_data`（已同步但尚未擷取）、或 `has_data`（有 summary）；
  - 回傳對應 `message`（尚未同步 vs 已同步但尚未擷取）。
- **前端**：`EmptyStateCard` 依 `dataStatus` 與 `message` 顯示：
  - **尚未同步帳號**：說明先到設定頁綁定、測試連線、點「立即同步帳號」，再回此頁點「更新資料」。
  - **尚未擷取數據**：說明已同步帳號，請點「更新資料」。
  - 避免在未同步或未擷取時讓使用者誤以為「廣告真的都是 0」。

## 6. Staging 驗收方式

1. **驗證狀態持久化**
   - 設定頁 → API 綁定 → 任填 FB 或 GA4 或 AI，點「測試連線」至成功或失敗。
   - 點「儲存所有設定」。
   - 重新整理頁面或離開再進入設定頁：對應區塊應顯示**上次**驗證結果（成功綠燈／失敗紅燈與訊息、時間），而非全部「尚未驗證」。

2. **欄位變更後驗證失效**
   - 在已驗證成功的欄位中修改一個字元（例如 token 尾端），點「儲存所有設定」。
   - 該區塊應變回「尚未驗證」；再改回原值並儲存，可再測試連線後確認會再次變為成功並持久化。

3. **同步狀態與按鈕**
   - 設定頁 API 綁定卡下方「資料同步狀態」：應顯示 Facebook / GA4 已同步數與最後同步時間（若從未同步則為「尚未同步」）。
   - 點「立即同步帳號」：應呼叫同步、toast 結果，數字與時間更新。

4. **Dashboard 空狀態**
   - 未同步時進入今日決策中心：應顯示「尚未同步帳號」及說明（設定頁綁定、同步、再更新資料）。
   - 已同步但未點過「更新資料」：應顯示「尚未擷取數據」及說明（點「更新資料」）。

5. **端對端（可選）**
   - 設定頁綁定 FB → 測試連線成功 → 儲存 → 立即同步帳號 → 到今日決策中心點「更新資料」；確認有 batch 後，摘要與數字不再為空狀態，且花費等不應全部為 0（若有實際廣告數據）。
