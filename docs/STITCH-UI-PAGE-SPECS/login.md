# 登入

## 1. 頁面用途

- **在做什麼**：未登入使用者輸入帳號密碼，通過後進入 `AuthenticatedApp`（側欄與主內容）。
- **主要給誰**：全角色（實際權限由後端與 `user.role` 決定，**待確認**細節）。
- **情境**：首次進入、session 過期後重新登入。

## 2. 進入方式

- **route**：未登入時由 `AppRouter` 顯示，無獨立 path 名稱（可視為應用根在登入狀態）。
- **從哪裡來**：直接開啟網址、點登出後。
- **前置條件**：後端 `/api/auth/login` 可用。

## 3. 首屏必須看到的內容

1. 產品名稱「AI 行銷總監」與副標文案。
2. 帳號、密碼輸入欄（含 `data-testid` 於 `login.tsx`）。
3. 登入送出按鈕。
4. 錯誤訊息區域（失敗時顯示）。

## 4. 區塊排列順序

- 單欄置中：品牌圖示與標題在上，表單卡片在中；若檔案含 dev 快速填帳按鈕，僅開發環境顯示（**待確認**與 build 設定一致）。

## 5. 按鈕盤點

### A. 主按鈕（Primary CTA）

- **登入**：提交表單，置於表單底部。

### B. 次按鈕（Secondary）

- 無必須次按鈕；dev 輔助填帳若有則屬開發用。

### C. 危險按鈕（Danger）

- 無。

### D. 輔助按鈕（Utility）

- **顯示／隱藏密碼**（圖示按鈕）：在密碼欄位旁。

## 6. 篩選 / 控制元件

- 無 tab、無日期；僅兩個文字欄位與密碼可見性切換。

## 7. 狀態資訊 / badge / warning

- 錯誤文案：帳密錯誤、伺服器錯誤、無法連線等，必須在表單附近可讀，不可被背景動畫淹沒。

## 8. 可收合但不能消失的區塊

- 錯誤訊息不可永久隱藏；帳密分欄不可合併成無標籤的單一輸入。

## 9. 此頁最不能被 AI 誤改的地方

- 登入與全屏 `LoadingScreen` 為不同狀態；表單可及性與錯誤回饋優先於裝飾動畫。

## 10. Stitch 用摘要

登入為單欄置中品牌加表單。首屏核心是帳號、密碼與登入主按鈕，密碼可切顯示為輔助。錯誤訊息須留在表單旁。視覺可加強信任與品牌，但不可移除欄位標籤或把流程改成隱藏步驟。無側欄、無診斷列。
