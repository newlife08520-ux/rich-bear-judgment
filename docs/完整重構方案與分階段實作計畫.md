# 完整重構方案與分階段實作計畫

本文件建立在**現有系統之上**，不重做、不推倒。目標產品：**素材管理 + 投放模板 + 批次建組 + 一鍵上稿**；正式儲存改為 **NAS**，本機僅作開發/fallback。

---

## A. 哪些現有能力直接沿用

| 項目 | 現狀 | 沿用方式 |
|------|------|----------|
| **AssetPackage** | shared/schema + asset-package-repository (JSON) | 結構不變；可新增欄位（如 productName 別名、解析用欄位）。 |
| **AssetVersion** | shared/schema + asset-version-repository (JSON) | 沿用；可新增「解析結果」欄位（見 E）。 |
| **版本上傳 API** | POST `/api/asset-packages/:id/versions/upload`，multer → upload-storage.saveFile | 沿用 route 與 multer；**僅把 saveFile 改為呼叫 Upload Provider 抽象**。 |
| **PublishDraft** | schema + publish-repository (JSON) + create/update/get/list | 沿用；**投放模板**可為獨立實體或「特殊草稿 + 標記」。 |
| **複製草稿** | 前端 openCopy(d) 預填表單 → POST 建立新草稿 | 邏輯沿用；擴充為「快速變體」（自動命名、只換版本）。 |
| **CRUD / routes / schema** | asset-packages, asset-versions, publish drafts 的 GET/POST/PUT/DELETE | 不刪除；僅在既有 route 上掛新 service（如命名解析、批次建組）。 |
| **Migration** | 現有 script（如 migrate-asset-to-package-version） | 若有 schema 欄位新增，可補小 migration 寫入預設值。 |

**不沿用 / 要替換的**：  
- **上傳儲存**：目前 `upload-storage.ts` 寫死 `.data/uploads`，改為透過 **Upload Provider** 介面，實作 local 與 NAS 兩種 provider。

## B. 哪些地方要重構

| 區塊 | 重構內容 | 說明 |
|------|----------|------|
| **上層流程：素材中心** | 流程改為「先上傳/先選版本，文案可後補」；廣告目的/狀態收進進階或拿掉 | 見先前「使用流程重整方案」；不重寫 schema，只改 UI 與必填邏輯。 |
| **上層流程：投放中心** | 三步：起點（新建/複製）→ 選素材包 → 選版本 + 投放設定；文案預設沿用、覆寫才展開；目標固定轉換 | 同上。 |
| **上層流程：批次建組** | 新增「批次建組器」：依解析結果或手動分組 → 預覽組數與命名 → 一鍵建立多筆草稿 | 新功能，見 F。 |

| **上傳儲存層** | 抽成 Upload Provider（local / NAS），不再寫死路徑 | 見 C、D。 |

**不重構的**：  
- AssetPackage / AssetVersion / PublishDraft 的**核心 schema 與 repository 介面**（僅擴充欄位或新增表/檔）。  
- 現有 API 路徑與權限模型（userId 隔離）。

## C. Upload Provider（local / NAS）怎麼設計

### 介面（抽象層）

```ts
// server/modules/asset/upload-provider-types.ts（或同目錄）

export interface SaveFileInput {
  userId: string;
  /** 可選：用於 NAS 路徑，例如 brand/product/period */
  context?: { brand?: string; product?: string; period?: string; packageId?: string };
  originalName: string;
  mimeType: string;
  source: Buffer | string; // Buffer 或本機暫存路徑
}

export interface SaveFileResult {
  fileUrl: string;   // 給前端/DB 存用的 URL（可為相對路徑或 NAS 對外 URL）
  fileName: string;  // 建議存進 DB 的檔名（顯示用）
  fileType: string;  // MIME
  /** 實體路徑，僅部分 provider 回傳（如 local 供 getFilePath 用） */
  storagePath?: string;
}

export interface IUploadProvider {
  saveFile(input: SaveFileInput): Promise<SaveFileResult> | SaveFileResult;
  /** 取得檔案實體路徑（供本機 GET 下載或 NAS 讀檔用）；不存在回傳 null */
  getFilePath(userId: string, filenameOrUrl: string): Promise<string | null> | string | null;
  /** 取得對外可存取 URL（前端 img/video src、Meta 上稿用）；若與 fileUrl 一致可直接回傳 */
  getPublicUrl(userId: string, fileUrlOrPath: string): Promise<string> | string;
}
```

- **saveFile**：取代現有 `upload-storage.saveFile` 的職責；回傳的 `fileUrl` 可為 `/api/uploads/{userId}/{filename}`（local）或 NAS 的 HTTP 路徑（若 NAS 有對外服務）。
- **getFilePath**：目前 routes 裡 `GET /api/uploads/:userId/:filename` 用 `getFilePath` 取實體路徑再 `res.sendFile`；改為向 provider 要路徑。
- **getPublicUrl**：本機開發多數直接回傳相對 URL；NAS 若走獨立網域或 CDN，可在此回傳完整 URL，供一鍵上稿時給 Meta 用。

### 實作

| Provider | 檔名建議 | 行為摘要 |
|----------|----------|----------|
| **Local** | `upload-provider-local.ts` | 維持現有邏輯：`.data/uploads/{userId}/{uuid_basename.ext}`，`fileUrl = /api/uploads/{userId}/{filename}`，`getFilePath` 回傳本機路徑，`getPublicUrl` 回傳相對或 baseURL + path。 |
| **NAS** | `upload-provider-nas.ts` | 依 D 的 NAS 路徑結構寫入；`saveFile` 回傳的 `fileUrl` 可為相對路徑（如 `/nas/...`）或 NAS 的 HTTP URL；`getFilePath` 回傳 NAS mount 的本機路徑；`getPublicUrl` 若 NAS 有對外 HTTP 則回傳該 URL。 |

### 組裝方式

- 設定檔或環境變數：`UPLOAD_PROVIDER=local | nas`，必要時 `NAS_BASE_PATH`、`NAS_PUBLIC_BASE_URL` 等。
- 在 **單一入口** 建立 provider 實例（例如 `server/modules/asset/upload-provider.ts`）：  
  `export const uploadProvider: IUploadProvider = process.env.UPLOAD_PROVIDER === 'nas' ? new NasUploadProvider(...) : new LocalUploadProvider();`
- **asset-package-routes** 的 `versions/upload`：改為呼叫 `uploadProvider.saveFile(...)`，不再直接呼叫現有 `uploadStorage.saveFile`。
- **routes.ts** 的 `GET /api/uploads/:userId/:filename`：改為 `uploadProvider.getFilePath(userId, filename)`，再 `sendFile`；若 provider 回傳 null 則 404。

這樣**不重寫** multer 與上傳 API，只替換「存到哪裡、路徑怎麼算」。

## D. NAS 路徑結構怎麼規劃

建議目錄結構（在 NAS 根目錄或 mount 點之下）：

```
{brandOrProduct}/           # 品牌或產品名，便於查找
  {userId}/                 # 使用者或部門 ID，權限/歸屬清楚
    {period}/               # 檔期或日期，例如 2025Q1、2025-03、雙12
      {packageId}/          # 素材包 ID，避免不同包混在一起
        {uniqueId}_{safeBasename}.{ext}   # 避免重名：uniqueId 可用 uuid 前 8 碼或時間戳
```

**範例**：  
`潔測泡泡/user_abc/2025-03/pkg_uuid_xyz/a1b2c3d4_主圖_9x16.png`

- **品牌/產品**：與「產品名稱」或素材包所屬一致，可由上傳 context 帶入或從 AssetPackage 查。
- **userId**：現有權限模型，直接沿用。
- **period**：可由上傳時帶入（例如表單選「檔期」），或預設為當月 `YYYY-MM`。
- **packageId**：上傳時已知（現有 API 即為 `POST .../packages/:id/versions/upload`），可寫入該層。
- **檔名**：`uniqueId` + 安全化的原始 basename + 副檔名，避免同包同檔期重名。

**查找**：  
- 依品牌/產品、userId、period 在 NAS 上瀏覽或備份。  
- 系統內仍以 `fileUrl`（或 NAS 的相對路徑）存於 AssetVersion，讀取時透過 provider 的 `getFilePath` / `getPublicUrl` 取得實體或對外 URL。

**注意**：NAS 若為 SMB/NFS mount，`getFilePath` 回傳的即為 mount 後的本機路徑；若 NAS 另提供 HTTP 服務，則 `getPublicUrl` 可組出對外 URL。

## E. 素材命名解析怎麼做

### 目標

從檔名（例：`潔測泡泡_A版`、`潔測泡泡_B版_9x16`、`藍色雙12_潔測泡泡_A版_4x5`）解析出：

- **檔期/活動前綴**（如「藍色雙12」）
- **主素材名稱**（如「潔測泡泡」）
- **版本代號**（如 A版、B版）
- **比例**（9x16、4x5、1:1 等）
- 其他可辨識資訊（可預留）

### 1. 最低可行版（MVP）

- **正則 + 關鍵字**：  
  - 比例：`/(\d+)[x:×](\d+)/i`、`/9x16|4x5|1x1|16x9/i` 等，正規化為 schema 的 `aspectRatio`（9:16, 4:5, 1:1, 16:9）。  
  - 版本代號：`/_?(A版|B版|C版|A|B|C)_/i` 或常見後綴。  
  - 檔期/活動：檔名開頭或第一段「非主名稱」可視為前綴（例如底線前一段）。  
  - 主名稱：剩餘最長一段或第二段（可依你們命名慣例微調）。
- **輸入**：上傳時的 `originalName` 或使用者手動輸入的「檔名」。
- **輸出**：寫入 AssetVersion 的**擴充欄位**（見下）。

### 2. 解析結果要存在什麼欄位

建議在 **AssetVersion**（或 shared schema）新增（可選，皆 optional）：

- `parsedCampaignPrefix?: string`  // 檔期/活動前綴
- `parsedAssetName?: string`      // 主素材名稱
- `parsedVariantCode?: string`   // 版本代號，如 A版、B
- `parsedAspectRatio?: string`   // 解析到的比例字串，若與 schema 的 aspectRatio 一致可只存 schema 欄位
- `parsedAt?: string`            // 解析時間，方便重跑

**不破壞現有欄位**：`fileName`、`assetType`、`aspectRatio` 仍為必填/既有；解析結果用於**建議** aspectRatio、預填 fileName、以及**批次建組時分組**。若解析出比例，可自動帶入 `aspectRatio` 下拉。

### 3. 解析失敗時怎麼處理

- **不擋上傳**：解析失敗時，`parsed*` 全不寫或寫 null；`aspectRatio` 由使用者手動選；`fileName` 沿用系統產生的或使用者輸入。
- **UI 提示**：列表或版本卡上可顯示「未解析出比例，請手動選擇」；建立版本表單仍強制選類型與比例。
- **日誌**：可記一筆 debug 或 metrics（哪個檔名解析失敗），供之後改規則用。

### 4. 怎麼讓使用者手動修正

- **版本編輯**：在「編輯素材版本」表單中，顯示解析結果欄位（檔期、主名稱、版本代號、比例）為**可編輯**；若為空則手動填。
- **批次建組前**：若依「主名稱+版本代號」分組，分組預覽畫面上可讓使用者合併/拆分組、改組名或單筆的 parsed 欄位，再送出建立。

## F. 批次建組器怎麼做

### 目標

- **不是**固定產 10 組，而是**依素材主體分幾組就產幾組**，或由使用者決定要建立哪些組。
- 例：A版 有 9:16 + 4:5 → 一組；B版 有 9:16 + 4:5 + 1:1 → 一組。

### 1. 現有資料模型如何延伸支援

- **不新增「組」實體**也可做：一組 = 一筆 **PublishDraft**，其 `assetPackageId` = 同一個素材包，`selectedVersionIds` = 該組內勾選的版本 ID。
- **分組依據**：依 AssetVersion 的 `parsedVariantCode` + `parsedAssetName`（或 `fileName`）做群組；同一組內再依 `aspectRatio` 列出可選版本。
- 若希望「組」有獨立名稱與模板關聯，可新增 **PublishTemplate**（見 G），一鍵建立時「每組一草稿 + 套用同一個模板」。

### 2. 使用者操作步驟

1. **選素材包**（或選「未分組的版本」來源）。
2. **系統依解析結果建議分組**：例如「A版」「B版」各一組，每組底下列出 9:16、4:5 等版本；使用者可合併/拆分/排除某些組。
3. **選投放模板**（或沿用預設）：帳號、預算、受眾、placement、命名規則等。
4. **預覽**：顯示「將建立 N 組」，每組名稱、使用的版本數、預覽命名（Campaign / Ad Set / Ad 名稱）。
5. **可手動改**：組數、組名、每組要含哪些版本、命名規則後綴（A/B/C 或 01/02）。
6. **一鍵建立**：後端依組數迴圈呼叫現有 `createDraft`，每筆草稿帶入模板 + 該組的 `selectedVersionIds` + 自動命名。

### 3. 系統如何預覽將建立幾組

- 後端提供 **POST /api/publish/batch-preview**（或 GET + query）：輸入 `packageId`、可選 `templateId`。
- 邏輯：取該包下所有版本 → 依 `parsedVariantCode`（或 fallback 檔名）分組 → 回傳 `{ groups: [{ variantCode, versionIds, suggestedName }, ...] }`。
- 前端依此顯示「將建立 3 組」與每組的版本清單、建議名稱；使用者可刪組、改名稱、勾選/取消版本。

### 4. 命名如何自動生成

- **命名規則**：來自**投放模板**（見 G）的 `campaignNamePattern`、`adSetNamePattern`、`adNamePattern`，例如 `{product}_{variant}_{date}`。
- 替換符：`{product}` = 素材包產品名，`{variant}` = 該組的版本代號（A版/B版），`{date}` = 當天或檔期，`{n}` = 序號 01/02。
- 一鍵建立時，每組帶入一組替換結果，寫入 PublishDraft 的 campaignName、adSetName、adName。

### 5. 如何允許使用者手動改數量 / 改名稱 / 排除某些組

- **預覽階段**：列表每組可「刪除組」「編輯組名」「勾選/取消版本」；改完再按「建立」才送 API。
- **API**：**POST /api/publish/batch-create** 接收 `{ templateId?, packageId, groups: [{ name?, versionIds: string[] }], namingOverrides?: { campaign?, adSet?, ad? } }`，後端依 groups 長度建立多筆草稿，命名優先用 namingOverrides 或模板規則。

## G. 投放模板怎麼做

### 投放模板應包含的欄位

- 廣告帳號、粉專、IG  
- 預算（每日/總計）、受眾、placement  
- 目標（預設轉換，可不顯示選項）  
- 預設文案、標題、CTA、網址（可選，用於新草稿預填）  
- **命名規則**：campaign / ad set / ad 的 pattern（如 `{product}_{variant}_0325`）

### 1. 現有 PublishDraft / 草稿複製能力哪些可沿用

- **複製草稿**：已能帶入整筆草稿的 accountId、pageId、igAccountId、預算、受眾、placement、命名、assetPackageId、selectedVersionIds 等；**模板**可視為「從一筆草稿另存為模板」或「從頭建一個只有設定、沒有選版本的草稿」。
- **createDraft**：可直接沿用；建立草稿時若傳入 `templateId`，先載入模板欄位再覆寫 assetPackageId、selectedVersionIds、campaignName 等。

### 2. 模板應新增哪些欄位

- 若模板 = **獨立實體**（建議）：
  - 表：**PublishTemplate**（或存成 JSON 檔 `publish-templates.json`，與現有 repo 風格一致）。
  - 欄位：id, userId, name, accountId, pageId?, igAccountId?, budgetDaily?, budgetTotal?, audienceStrategy, placementStrategy, campaignObjective（固定「轉換」）, primaryCopy?, headline?, cta?, landingPageUrl?, campaignNamePattern?, adSetNamePattern?, adNamePattern?, createdAt, updatedAt。
- 若模板 = **特殊草稿**：在 PublishDraft 加 `isTemplate: boolean`，列出草稿時篩選 `isTemplate === true` 當模板列表；建立草稿時「從模板建立」= 複製該筆草稿但清空 selectedVersionIds、再讓使用者選包與版本。  
  **建議**：獨立 **PublishTemplate** 較乾淨，不污染草稿列表。

### 3. 模板與素材包、素材組之間怎麼關聯

- **無直接 FK**：模板不綁定素材包；建立草稿時「選模板 + 選素材包 + 選版本（或批次選組）」。
- **關聯僅在建立當下**：一鍵建立時，每組草稿 = 同一 templateId + 同一 packageId + 該組的 versionIds + 該組的命名替換結果。

## H. 分階段實作計畫

### 階段 0：前置（不影響現有行為）

- **上傳抽象層**：新增 `IUploadProvider`、`LocalUploadProvider`（把現有 upload-storage 邏輯搬進去）、在 asset-package-routes 改為呼叫 provider；環境變數預設 `local`。  
- **改動檔案**：新增 `upload-provider-types.ts`、`upload-provider-local.ts`，重構 `upload-storage.ts` 為呼叫 local provider（或直接改為 local provider 實作）；`asset-package-routes.ts` 改一行呼叫；`routes.ts` 的 GET uploads 改為用 provider.getFilePath。  
- **新增欄位**：無。  
- **風險**：路徑或 URL 行為不一致；驗收：上傳一檔 → 版本可建立 → 前台可顯示圖片/影片。  
- **驗收**：本機上傳、列表與編輯流程與現在一致，僅程式結構改變。

---

### 階段 1：NAS Provider + 路徑結構

- **實作**：`upload-provider-nas.ts`，依 D 的目錄結構寫入；設定 `NAS_BASE_PATH`、可選 `NAS_PUBLIC_BASE_URL`；`UPLOAD_PROVIDER=nas` 時啟用。  
- **改動檔案**：新增 `upload-provider-nas.ts`，`upload-provider.ts` 依 env 選擇 provider；saveFile 的 context 從 route 傳入（userId, packageId 已有；period/brand 可從 body 或 package 查）。  
- **新增欄位**：無（或 AssetPackage 加 `period?` 供 NAS 路徑用）。  
- **風險**：NAS mount 權限、路徑長度、編碼；驗收：NAS 上目錄與檔名符合規劃、GET 可透過 getFilePath 讀到（若 NAS 用 mount）。  
- **驗收**：設定 NAS 後上傳，檔案出現在 NAS 正確路徑；本機改回 local 仍可上傳。

---

### 階段 2：素材命名解析（MVP）

- **實作**：`parseAssetFileName(originalName: string)` 回傳 `{ campaignPrefix?, assetName?, variantCode?, aspectRatio? }`；上傳或建立版本時呼叫，將結果寫入 AssetVersion。  
- **改動檔案**：新增 `server/modules/asset/asset-name-parser.ts`；asset-version-service 在 create 時呼叫 parser，寫入擴充欄位；shared schema 的 AssetVersion 新增 optional parsed* 欄位。  
- **新增欄位**：AssetVersion：parsedCampaignPrefix?, parsedAssetName?, parsedVariantCode?, parsedAt?；aspectRatio 仍由使用者選，parser 可建議。  
- **風險**：命名規則多樣，解析率未必 100%；驗收：幾組範例檔名解析正確、失敗不擋建立。  
- **驗收**：上傳「潔測泡泡_B版_9x16.png」→ 版本列表顯示解析出 B版、9:16；編輯可改解析結果。

---

### 階段 3：素材中心流程重整（上層）

- **實作**：第一屏「名稱 + 產品名稱 + 素材版本區」優先，文案區可後補、摺疊；廣告目的/狀態進階或隱藏；建立時不擋文案必填。  
- **改動檔案**：`client/src/pages/assets.tsx`（或素材中心相關元件）；後端 schema 若允許文案空字串則不改，否則放寬驗證。  
- **新增欄位**：無（或 AssetPackage 的 brandProductName 對應 UI 改為「產品名稱」）。  
- **風險**：現有必填習慣改變；驗收：能先建包、只上傳版本再補文案。  
- **驗收**：新建素材包 → 只填名稱與產品名 → 上傳一版 → 儲存成功；再補文案儲存。

---

### 階段 4：投放模板實體與 API

- **實作**：PublishTemplate 的 schema + repository（JSON 或 DB）；CRUD API；投放中心「從模板建立」入口。  
- **改動檔案**：新增 shared 的 PublishTemplate 型別、publish-template-repository、publish-template routes；publish-placeholder 頁面可選模板。  
- **新增欄位**：PublishTemplate 全欄位（見 G）。  
- **風險**：與現有草稿複製重疊；驗收：可建立/編輯/刪除模板、從模板建立草稿會帶入設定。  
- **驗收**：建立模板「轉換標準」→ 從模板建立 → 草稿預填帳號/預算/受眾/placement，再選素材包與版本。

---

### 階段 5：投放中心三步 + 覆寫摺疊 + 目標固定

- **實作**：Step 1 選「新建 / 從上組複製」；Step 2 選素材包；Step 3 選版本 + 投放欄位；文案區預設摺疊、顯示「沿用素材包」摘要；廣告目標不顯示選項、固定轉換。  
- **改動檔案**：`client/src/pages/publish-placeholder.tsx`。  
- **新增欄位**：無。  
- **風險**：使用者找不到目標選項；驗收：主流程三點完成、文案可展開覆寫、複製後自動命名（見階段 6）。  
- **驗收**：選複製 → 選包 → 選版本 → 只填預算/命名 → 建立；文案未展開時顯示沿用摘要。

---

### 階段 6：複製 = 快速變體（自動命名、只換版本）

- **實作**：複製時名稱自動加後綴（A/B/C 或 01/02）；可選「沿用同一包、只重選版本」；後端若需可提供 POST /api/publish/drafts/from-copy 或前端沿用現有 POST 帶入 name 後綴。  
- **改動檔案**：publish-placeholder 的 openCopy 與表單預設；可選後端 createDraft 支援 nameSuffix 或 pattern。  
- **新增欄位**：無（命名在 create 時由前端或後端規則產生）。  
- **風險**：命名衝突；驗收：複製兩次得到「原名 A」「原名 B」，只改版本即可送出。  
- **驗收**：複製草稿 → 表單預填、名稱為「XXX 02」→ 只改版本 → 建立。

---

### 階段 7：批次建組器（預覽 + 一鍵建立）

- **實作**：後端 batch-preview、batch-create；前端「批次建組」頁或區塊：選包 → 顯示分組預覽 → 可調組/改命名 → 一鍵建立多筆草稿。  
- **改動檔案**：新增 publish 的 batch-service、batch routes；前端新增批次建組 UI。  
- **新增欄位**：無（或 template 的 naming pattern 欄位若尚未做則在階段 4 一併做）。  
- **風險**：一次建立大量草稿、命名重複；驗收：預覽組數正確、建立後列表有 N 筆、每筆對應一組版本。  
- **驗收**：選包「潔測泡泡」→ 系統建議 A版/B版 兩組 → 預覽 2 組 → 一鍵建立 → 草稿列表出現 2 筆，各對應 A 版與 B 版版本。

---

### 階段 8：一鍵送 Meta（若本輪要做）

- **實作**：呼叫 Meta API 建立 Campaign / Ad Set / Ad；使用草稿內 accountId、創意、預算等。  
- **改動檔案**：新增 Meta 發佈服務、publish 的「發佈」按鈕與 API。  
- **風險**：Token、權限、限流；驗收：選一筆草稿 → 一鍵發佈 → Meta 後台可見。  
- **備註**：你已說「之後再一鍵送 Meta」，此階段可列在計畫中延後實作。

## I. 自我檢討：最可能做錯的 5 件事

1. **又做成「工程表單」**  
   - 每個畫面塞滿欄位、必填一堆，投手要點很多次才能完成一件事。  
   - **避免**：嚴格遵守「先上傳/先選版本，文案可後補」「文案預設沿用、覆寫才展開」「目標固定轉換不給選」；主流程點擊數與必填項在設計時寫死上限（例如建立一筆草稿 ≤ 5 步）。

2. **命名解析過度設計**  
   - 一開始就做複雜 NLP 或多語系，導致延遲上線、解析率卻未必好。  
   - **避免**：MVP 只做正則 + 關鍵字，明確列出 3～5 種你們真實檔名範例，只針對這些寫規則；其餘「未解析」就手動選，不擋流程。

3. **批次建組與「組」的概念混淆**  
   - 引入新實體「AssetGroup」或複雜關聯，導致與現有 PublishDraft、AssetVersion 不同步。  
   - **避免**：一組 = 一筆 PublishDraft（同 package、一組 versionIds）；不新增組表，只新增「預覽 + 一鍵建立多筆草稿」的流程與 API。

4. **上傳儲存重寫掉現有邏輯**  
   - 直接改寫 upload-storage 成 NAS only，本機開發或沒掛 NAS 時掛掉。  
   - **避免**：先抽 Provider、local 實作與現有行為完全一致；NAS 為第二個實作；用 env 切換，預設 local。

5. **投放模板與草稿複製重複做兩套**  
   - 模板一套、複製草稿另一套，命名與欄位不一致，投手困惑。  
   - **避免**：模板 = 只存「設定 + 命名規則」、不存選中的版本；「從模板建立」= 載入設定後再選包與版本；「複製草稿」= 載入整筆（含版本）+ 自動改名 + 可只換版本。兩者在 UI 上可並列為「從模板建立」「從上組複製」，底層共用 createDraft 與命名規則。

---

以上為完整重構方案與分階段實作計畫，建立在現有 AssetPackage、AssetVersion、PublishDraft、上傳 API、複製草稿與 CRUD 之上；僅重做上層流程、上傳儲存抽象、命名解析、批次建組與投放模板，並分階段驗收與控險。  
實作前建議先確認：NAS 實際 mount 方式與有無 HTTP 對外、命名範例清單、以及模板與複製的並存方式，再從階段 0 開始執行。
