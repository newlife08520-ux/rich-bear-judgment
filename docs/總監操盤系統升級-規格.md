# 總監操盤系統升級 — 系統規格與落地對照

本文件將「真正能操盤的總監系統」理念轉為**可實作的系統規格**：頁面怎麼排、欄位怎麼改、判斷怎麼做、哪些檔要動。交付後依此規格分階段實作，commit hash 於各階段完成時補上。

**本規格分兩層：上層是靈魂（產品使命、總監思維、介面節奏），下層是工程落地（API、欄位、區塊）。實作時必須先服從靈魂，再填技術清單。**

---

## 靈魂層：產品定位與使命（規格最上位，實作時必須遵守）

### 產品定位（一句話）

> **這是一套以商品為中心、以利潤為目的、以 8:2 為優先序、兼具老闆／投手／行銷總監思維的操盤系統。**

更白話：**它不是告訴你「數字發生了什麼」，而是告訴你「現在該做哪件事最賺」。**

### 產品使命（寫進規格最前面，不是散落在各頁）

本系統的任務**不是呈現所有數據**，而是**幫公司做出最有價值的少數決策**。

- 優先放大最值得放大的商品與素材  
- 優先止血最危險、影響最大的高預算項目  
- 不誤殺延遲型好貨與可救商品  
- 不讓真正有潛力的黑馬素材被埋沒  
- 不讓團隊浪費時間處理低影響的小事  
- 在可接受風險下，幫公司賺更多、虧更少  

### 系統總原則（靈魂，驅動所有邏輯與介面）

| 原則 | 內容 |
|------|------|
| **原則 A：先看影響力，再看好壞** | 不是先處理最爛的，而是先處理「最爛且最有影響的」「最好且最值得放大的」。 |
| **原則 B：先商品，再素材，再帳號** | 帳號不是主角，商品才是主角。帳號是容器，素材是商品的槓桿。 |
| **原則 C：8:2 分層落地** | 全公司層、帳號層、商品層、素材層，每一層都做 8:2，不是只做一次。 |
| **原則 D：動作比報表重要** | 系統最終要輸出的是：放大、止血、觀察、不要誤殺、值得延伸、先補素材、調目標成果——而不是一堆分數卡。 |
| **原則 E：一天最多調整 2 次** | 穩定時 1 次就夠。系統不是煽動你亂調，而是幫你節制地調。 |

### 介面層的靈魂（不要只是邏輯正確，要讓人一眼懂）

- **要求 1**：首頁第一屏只放高價值區塊，不要塞一堆帳號報表。  
- **要求 2**：危險用紅、機會用綠、觀察用黃，但不要滿版都是顏色。  
- **要求 3**：每一張卡都要有一句「總監判語」，不是只有數字。  
- **要求 4**：每個表都要先顯示動作與優先級，再顯示完整數據。  
- **要求 5**：介面本身也要符合 8:2——最重要的資訊最大、最先、最清楚。  

### 規格評價與補齊說明

本規格**可以做為骨架**，但若只有 API／欄位／區塊清單，會比較像「很厲害的工程規格」而不是「總監系統」。因此已在上方補齊三種靈魂：

1. **產品使命**：寫在最前面，驅動所有決策與介面——幫公司賺最多、虧最少、先做影響最大的事、不誤殺不埋沒、不浪費時間在低影響小事。  
2. **總監語言**：總監判語（每張卡一句話）、真好/假好/可救/真爛全系統共通、動作優先（放大/止血/觀察/不要誤殺/延伸）、給投手/給設計一句話。  
3. **介面節奏**：首頁第一屏只放高價值區塊、介面符合 8:2、危險/機會/觀察用色節制、表先顯示動作與優先級再顯示完整數據。  

實作時：**先對齊靈魂，再對齊技術清單。**

## 一、總核心：分層 8:2 落地規格

### 1.1 資料與維度

| 層級 | 現有資料來源 | 規格要求 | 實作要點 |
|------|--------------|----------|----------|
| **全公司** | `CrossAccountSummary`、`AnalysisBatch` | 全公司 8:2：誰在賺、誰在燒、誰是主力 | 已有 `boards`（danger/opportunity/scale/leakage）；需改為「影響力加權」排序，見 §二 |
| **帳號** | `AccountHealthScore[]`、`CampaignMetrics` by account | 帳號內 8:2：該帳號下哪幾個商品/活動影響最大、最危險、最值得放大 | 新增「帳號維度」的 product-level 與 creative-level 聚合；API 回傳 `accountId` + 該帳號內商品/素材排名 |
| **商品** | `ProductLevelMetrics`（tag-aggregation）、`CreativeLeaderboardRow` | 商品內 8:2：該商品下哪幾支素材在撐、哪幾支在拖、黑馬、疲乏 | 擴充 `aggregateByProduct` / creative 聚合，產出「商品內素材排名」+ 標籤（主力/拖累/黑馬/疲乏） |
| **素材** | Ad/Creative 維度、`CreativeLeaderboardRow`、`materialTier` | 素材內 8:2（同商品多素材比較）；單素材 1d/3d/7d、漏斗、疲乏 | 現有 `CreativeLeaderboardRow`、`materialTier`；需補「所屬商品」「幫/拖」「whyNotMore」「建議幅度」 |

### 1.2 影響力權重（先處理影響大的）

以下權重寫入**決策排序**，不是取代 ROAS/保本，而是與「好/爛程度」一起決定優先順序：

- **花費佔比**：該實體花費 / 所屬上層總花費（帳號或全公司）
- **營收佔比**：同上
- **是否主力商品**：依全公司商品花費或營收排名，前 N 為主力（可設 20% 或前 5）
- **是否高預算運行中**：例如日預算或總花費 > 門檻（如 5000/10000）
- **是否在關鍵帳號**：帳號健康度差或帳號花費佔比高

**實作**：在 `scoring-engine` / `real-data-transformers` 或新模組 `director-priority-engine` 中，為每個候選（活動/商品/素材）計算 `impactScore`，並在排序時使用 `impactScore * urgency` 或 `impactScore * opportunity`，使「又爛又影響大」與「又好又影響大」排到最前。

## 二、真好 / 假好 / 可救 / 真爛 落地

### 2.1 定義（寫入規格與引擎）

| 標籤 | 條件（邏輯） | 建議動作方向 |
|------|--------------|--------------|
| **真好** | 利潤結構 OK、ROAS 遠高於保本/目標、1d/3d/7d 都穩、漏斗比例健康、放大後無明顯掉速、高預算仍守住 | 可放大、守位；納入「今日最該放大」 |
| **假好** | 僅單日好看、樣本小、可能運氣出單、一放大易崩 | 觀察、小試探加碼、不建議大拉 |
| **可救** | 商品/方向可能對，但素材疲乏、角度老、頁面或受眾疲乏 | 修/換/補（換素材、收緊受眾、優化頁面），不先砍 |
| **真爛** | 高花費、過不了保本、多窗口都差、漏斗差、結構性問題 | 降/停；納入「今日最該止血」 |

### 2.2 與現有欄位對應

- **現有**：`riskLevel`（danger/warning/watch/stable/potential）、`scoring.diagnosis`、`scoring.recommendedAction`、`roiLabel`（Winner/STABLE/Lucky/FunnelWeak/Retired）、`lifecycleStage`。
- **擴充**：新增 `qualityLabel: "真好" | "假好" | "可救" | "真爛"`（或 `directorQualityLabel`），由引擎依上述條件寫入；可放在 `CampaignMetrics` 擴充或 `ScoringResult` 擴充，或新介面 `DirectorCampaignVerdict`。
- **實作檔**：`shared/schema.ts`（型別）、`server/scoring-engine.ts` 或新檔 `server/director-quality-engine.ts`（計算 qualityLabel）、`server/real-data-transformers.ts`（組進 API 回傳）。

### 2.3 商品層 / 素材層

- **商品**：依該商品底下活動/素材彙總後的 ROAS、多窗口、漏斗、花費，給該商品一個 `qualityLabel`（真好/假好/可救/真爛）。
- **素材**：依該素材（或 ad/creative 維度）的 ROAS、窗口、漏斗、疲乏、樣本數，給 `qualityLabel`；並標「幫商品 / 拖累商品 / 黑馬 / 疲乏」。

### 2.4 真好／假好／可救／真爛：全系統共通語言

這四個分類**不是只在後台算完就收起來**，必須**出現在**：

- 首頁（今日操盤室）
- 商品作戰室
- 素材作戰台
- 今日最該動的 5 件事（每筆建議都帶品質判讀）
- 建議動作引擎（輸出時用同一套語言）

這樣整個團隊才會講同一種語言；實作時所有相關 API、卡片、表格都要能顯示或篩選 `qualityLabel`。

## 三、高效 / 目標成果型廣告納入判斷

### 3.1 辨識

- **輸入**：需從 Meta 或現有結構取得「是否為目標成果型（cost cap / ROAS goal / CPA goal）」及「目標值」；若 API 無，可先由 campaign 名稱或備註欄位推測，或由設定檔對應。
- **輸出**：每筆活動/廣告集標記 `biddingType: "standard_budget" | "cost_cap" | "roas_goal" | "cpa_goal"` 及 `targetOutcomeValue`（若有）。

### 3.2 動作層（三種建議）

| 情境 | 建議 |
|------|------|
| 預算吃不滿、比例好、利潤厚、有擴量空間 | **調高目標成果 / 成本門檻**（放寬拿量） |
| 花得出去但品質差、比例掉、要守 ROAS/獲利 | **調低目標成果 / 收緊門檻** |
| 穩定跑、花費與比例可接受 | **維持目標成果不動，只觀察** |

### 3.3 實作

- **Schema**：`CampaignMetrics` 或 Ad Set 層級擴充 `biddingType`、`targetOutcomeValue`（可選）。
- **後端**：在 `meta-data-fetcher` 或分析 pipeline 中盡可能帶入 bidding 類型；若無則預設 `standard_budget`。
- **決策引擎**：若 `biddingType !== "standard_budget"`，建議動作不只有加/減預算，而要產出「調高/調低目標成果」或「維持觀察」；在 `buildRealBudgetRecommendations` 或新函式中分支處理。

### 3.4 靈魂：系統要比一般報表更懂

高效／目標成果型廣告**不能只看「加預算／減預算」**。系統要能區分：

- **量拿不到**：預算吃不滿、比例很好 → 考慮調高目標成果（放寬拿量）。
- **量拿到了但變差**：花得出去但品質差、比例掉 → 考慮調低目標成果（收緊門檻）。
- **目前很穩** → 維持，不亂動。

也就是：**「量拿不到」和「量拿到了但變差」是不同問題，要給不同建議。** 實作時決策分支必須分開處理這兩種情境。

## 四、動態拉升 / 降速階梯

### 4.1 拉升階梯（概念）

- **試探加碼**：剛過門檻，先小拉。
- **中幅拉升**：證明穩定後，進第二階段。
- **大幅拉升**：高利潤、高穩定、高信心時可大拉。
- **高預算守位**：到 3 萬/5 萬/10 萬級，重點守可接受掉速，不暴拉。

### 4.2 降速階梯

- **小降觀察**、**中降風險控制**、**大降止血**、**關停**。

### 4.3 實作

- **現有**：`scale-score-engine` 已有 `BudgetRecommendation`（先降/小降觀察/維持/可加碼/高潛延伸）、`getBudgetRecommendation`、`getScaleBudgetRecommendation`。
- **擴充**：
  - 建議幅度不只「百分比」，可產出 **階梯標籤**（試探加碼/中幅拉升/大幅拉升/守位；小降/中降/大降/關停）與 **建議區間**（如 +10%~20% 或 -20%~-30%），寫入 `BudgetRecommendation` 或新欄位 `suggestedTier`、`suggestedRange`。
  - 輸入納入：影響力、商品利潤、多窗口穩定度、漏斗質量、素材疲乏、**今日已調整次數**（見 §五）。

## 五、「一天最多調整 2 次」寫進系統

### 5.1 資料

- **儲存**：每個實體（活動 / 廣告集 / 或依你們最小調整單位）需有「今日已調整次數」。
- **實作**：在 `server/campaign-decisions-store.ts` 或新 store 中，以 `entityId` + `date (YYYY-MM-DD)` 為 key，記錄當日調整次數；每次使用者執行「加預算/減預算/調目標成果」時 +1（或由前端呼叫一 POST 端點寫入）。

### 5.2 決策邏輯

- 若 **今日已調整 ≥ 2 次**：該實體建議降級為「優先觀察」「除非重大風險不建議再動」；或建議改為「今日第二次再觀察後執行」「暫不動只追蹤」。
- 若 **高預算且運行穩定**：不建議頻繁加減，傾向「維持觀察」。

### 5.3 建議動作分級（要成為系統硬規則）

每筆建議都要考慮：今日已調幾次、上次調整時間、是否仍在觀察窗、是否高預算關鍵項。然後把建議分成：

- **可立即動**：高影響 + 高風險/高機會 + 今日調整次數 < 2。
- **晚點再動**：今日第二次再觀察後執行；重要但非緊急，或今日已 1 次。
- **今日不建議再動**：今日已 2 次，或穩定無明顯理由；系統明確標示「今日已達上限，建議觀察」。

不只是 UI 提醒，而是**進決策引擎**——建議產出時就帶上這三級之一，前端依此顯示樣式與文案。

### 5.4 API / UI

- **GET** 某實體或全公司「今日已調整次數」：供首頁與各作戰室顯示。
- **POST** 記錄一次調整（可與現有「儲存決策」整合）。
- **UI**：每筆建議旁顯示「今日已動 N 次」；若 N ≥ 2 顯示「今日已達上限，建議觀察」。

## 六、首頁 / 今日操盤室 怎麼排

**靈魂要求：首頁必須一打開就像總監晨會桌面。第一眼一定先看到「今天最該放大什麼、最該止血什麼、不要誤殺什麼、最值得延伸什麼」——行動優先，不是資訊架構優先。**

### 6.1 首頁五區（依優先序，介面也要符合 8:2）

| 順序 | 區塊 | 內容與靈魂 |
|------|------|------------|
| **第一區** | **今日最該動的 5 件事** | 整頁靈魂。每一筆要有：**類型**（放大／止血／不要誤殺／延伸／補素材）、**對象**（商品／商品+素材／帳號+商品）、**理由**（一句人話）、**建議動作**（調多少／做什麼）、**今日已調整次數**、**風險條件**。每張卡都要有一句**總監判語**，不是只有數字。 |
| **第二區** | **主力商品戰情** | 只看主力商品，不看全部商品。表格或卡片：商品名、花費、營收、ROAS、1d/3d/7d、品質標籤（真好/假好/可救/真爛）、所屬帳號。 |
| **第三區** | **高預算危險商品** | 只看真正高影響風險，不看邊角料。依影響力排序；每筆帶帳號、商品、建議動作、今日已動次數、總監一句話。 |
| **第四區** | **黑馬素材** | 只放容易被忽略、但對主力商品有高價值的素材。花費不高但潛力高、未達今日 2 次。 |
| **第五區** | **今日操作節制提醒** | 提醒哪些已調過、哪些今天最好不要再亂動。例如：「以下 N 項今日已調整 2 次，建議觀察」「以下 M 項高預算穩定運行，暫不建議動」。 |

### 6.2 今日最該動的 5 件事 — 每筆必備欄位

- **類型**：放大 / 止血 / 不要誤殺 / 延伸 / 補素材  
- **對象**：商品 / 商品+素材 / 帳號+商品  
- **理由**：一句人話（總監判語）  
- **建議動作**：調多少 / 做什麼  
- **今日已調整次數**  
- **風險條件**（若為止血或觀察）

## 八、商品作戰室 改成商品主戰場

**靈魂要求：商品作戰室不是商品報表，是「商品主戰場」。每個商品都要能一眼回答：值不值得繼續砸、為什麼值得／為什麼危險、靠哪些素材在賺、被哪些素材在拖、現在最該做什麼。**

### 8.1 商品層必備欄位

| 欄位 | 說明 |
|------|------|
| 商品 | 商品名 |
| 帳號歸屬 | 所屬廣告帳號 |
| 花費、營收、ROAS | 基本成效 |
| 成本比狀態、保本 ROAS、目標 ROAS、Headroom | 利潤結構 |
| 1d / 3d / 7d | 多窗口 |
| ATC / Purchase / Ratio | 漏斗比例 |
| 真好 / 假好 / 可救 / 真爛 | 品質標籤（全系統共通語言） |
| 建議動作、今日已動次數 | 行動與節制 |

### 8.2 商品詳情頁四區

1. **總監一句判語**：該商品目前是「真好可放大」「假好先觀察」「可救先換素材」「真爛先止血」等人話結論，不是只有數字。
2. **在撐的素材**：該商品底下正在幫賺的素材列表（幫/主力/黑馬），含縮圖、花費、ROAS、一句話。
3. **在拖的素材**：該商品底下在拖累的素材（拖/疲乏/該停），含同上。
4. **下一步建議**：建議動作、建議幅度、今日已動次數、節制提醒。

### 8.3 區塊與實作

- 商品總覽表（含上述必備欄位；排序先動作與優先級，再完整數據）。
- 點進單一商品 → 詳情頁四區（判語、在撐的素材、在拖的素材、下一步建議）。
- 建議動作清單（含一天 2 次節制邏輯）。
- **實作**：可沿用或擴充 `client/src/pages/products.tsx`，或新頁 `product-war-room`；資料由 product 聚合 API + 品質標籤 + 素材幫/拖標籤組成。

## 九、素材作戰台

**靈魂要求：素材頁要回答——這支素材在幫哪個商品？幫到什麼程度？它是不是黑馬？是不是疲乏？值得延伸還是該停？每支卡都要有「給投手一句話」「給設計一句話」，不是只有數字。**

### 9.1 每支素材卡必備

| 欄位 | 說明 |
|------|------|
| 素材縮圖 | 視覺辨識 |
| 所屬商品、所屬帳號 | 歸屬 |
| 花費、營收、ROAS | 成效 |
| Creative Edge、1d/3d/7d | 潛力與多窗口 |
| 疲乏程度、Ratio | 漏斗與疲乏 |
| **定位** | 主力 / 黑馬 / 觀察 / 疲乏 / 該停（與真好/假好/可救/真爛對齊） |
| **給投手一句話** | 總監判語：該放大、該停、該換、該觀察等 |
| **給設計一句話** | 借鑑點：可延伸方向、該避開的疲乏訊號等 |

### 9.2 區塊與篩選

- 素材列表/卡片：上述必備欄位；每張卡一句總監判語。
- 篩選：黑馬、高預算爛素材、疲乏、可延伸。
- 排序：先動作與優先級，再完整數據；可依影響力×急迫或機會。
- **實作**：可擴充 `client/src/pages/creative-lifecycle.tsx` 或 fb-ads 創意相關區塊；後端需提供 creative 維度 + 所屬商品 + 幫/拖 + 品質標籤 + 給投手/給設計文案（可由引擎或 AI 產出）。

## 十四、實作優先順序與 Commit

**規格定位**：本文件可作為**骨架**；已補上三種靈魂——**產品使命**、**總監語言**（判語、共通語言、動作優先）、**介面節奏**（第一屏只放高價值、介面符合 8:2）。實作時先服從靈魂，再填技術清單。

### 建議實作順序（不要一口氣全做）

| Phase | 內容 | 目標 |
|-------|------|------|
| **Phase 1** | 首頁今日操盤室重做 | 第一屏做好，一打開就覺得「真的變了」：今日最該動 5 件事、主力、高預算危險、黑馬、節制提醒；每張卡有總監判語。 |
| **Phase 2** | 商品作戰室重做 | 商品主戰場：總監一句判語、在撐的素材、在拖的素材、下一步建議；商品層必備欄位與品質標籤。 |
| **Phase 3** | 素材作戰台 | 黑馬、疲乏、幫/拖做出來；每支卡：給投手一句話、給設計一句話、定位（主力/黑馬/觀察/疲乏/該停）。 |
| **Phase 4** | 高效廣告／目標成果型廣告邏輯 | 辨識 biddingType；量拿不到 vs 量拿到了但變差 分開建議；調高/調低目標成果、維持觀察。 |
| **Phase 5** | 一天最多調 2 次與觀察期 | 今日調整次數 store、決策引擎硬規則、可立即動/晚點再動/今日不建議再動；系統成熟、不躁進。 |

技術依賴可並行或微調：例如影響力權重與真好/假好/可救/真爛引擎、Schema 擴充，可在 Phase 1 前或 Phase 1 內一併做；今日調整次數可在 Phase 1 就接入，Phase 5 再強化邏輯。

**Commit hash**：待各階段實作完成後補上。
