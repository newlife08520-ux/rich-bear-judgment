## 4. 後端防呆規則怎麼做

- **時機**：POST `/api/workbench/prompts/:mode/draft` 儲存草稿前、POST `/api/workbench/prompts/:mode/publish` 發布前（先讀取當前 draft 內容再驗證）。
- **邏輯**：呼叫 `validateOverlayContent(content)`。若 `ok === false`，回傳 HTTP 400，body 含 `message`（reason）、`errorCode: "OVERLAY_PERSONA_BLOCKED"`、`matchedLabel`（命中的高風險標籤）。
- **實作位置**：`server/prompt-overlay-validation.ts` 以正則清單比對；`server/routes.ts` 在 draft 與 publish 兩處呼叫並依結果決定是否寫入／發布。

## 6. 單次 Content Judgment API 如何改成 audit 一致

- **路徑**：`POST /api/content-judgment/start`（`server/routes.ts`）。
- **變更**：
  1. **輸入充足度**：在組裝 prompt 前檢查 `input.url` 或 `input.text` 長度。若無 URL 且 `text.trim().length < 30`，回傳 400，`message`：「請提供 URL 或至少 30 字以上內容再審…」，`errorCode: "INPUT_INSUFFICIENT_FOR_AUDIT"`。
  2. **明確傳 workflow**：`getAssembledSystemPrompt({ ..., workflow: "audit" })`，不再依預設 clarify。
  3. **行為一致**：因 `workflow: "audit"`，組裝會自動加上 MODE A/B/C/D（依 judgmentType / uiMode）、`STRUCTURED_JUDGMENT_OUTPUT_INSTRUCTION`，與 chat 的 audit 路徑一致。
- **日誌**：`console.log` 增加 `workflow=audit`。

## 7. uiMode / 視角 / audit mode profile 如何拆責任

- **視角（view role）**  
  - 對應欄位：仍為 `uiMode`（boss / buyer / creative）。  
  - 責任：選取「哪一份已發布 Overlay」、該視角下的輸出偏向與呈現優先順序。  
  - 程式位置：`getPublishedPrompt(effectiveMode)` 取該視角之已發布內容；組裝時以 `customMainPrompt`（publishedOverlay）疊在 Core 之後。  
  - 註解：在 `AssembleOptions` 與組裝註解中寫明「視角（view role）」「角色視角補充 Overlay」。

- **audit mode profile**  
  - 對應：同一 `uiMode`，但**僅在 `workflow === "audit"` 時**參與。  
  - 責任：決定要疊哪些審判骨架（MODE A/B/C/D），與 judgmentType 的對應。  
  - 程式位置：`rich-bear-prompt-assembly.ts` 內 `UI_MODE_TO_INTERNAL`，註解為「audit mode profile：僅在 workflow=audit 時使用」；`getAssembledSystemPrompt` 中 `if (isAudit)` 時才依 judgmentType 或 uiMode 取 internalModes 並疊 MODE 文案。

- **前端**：仍沿用 `uiMode` 欄位名稱；內部邏輯與註解已區分「視角」與「audit mode profile」。

## 8. 哪些東西明確保留在數據引擎，不再進 prompt

以下維持由 **scoring-engine / analysis-engine / routes 內計算**，**不**寫入人格或 Calibration 內文，僅以 **Data Context（Layer 4）結果字串** 進 prompt：

- 8:2 法則相關計算與建議
- 投手公式（預算、擴量、rescue 等）之計算
- Scale Readiness、趨勢與漏斗判斷
- 預算建議、suggestedAction、suggestedPct、reason、whyNotMore
- 各項 ROAS、成本比、保本、ATC、轉換等指標

上述僅以 `buildDataContextSection(context)` 產出之「本任務資料」字串進入組裝，**公式與規則本身**不放入 Core / Overlay / Calibration。

## 9. 驗收方式

1. **Prompt 設定頁**：進入「角色視角 Overlay」頁，確認標題與說明皆為視角補充語意，無「主 prompt／人格編輯」字樣。
2. **防呆**：在 Overlay 草稿中輸入「你的最高任務是…」或「角色核心」等，儲存或發布應被阻擋，並顯示後端回傳之錯誤訊息。
3. **單次審判**：呼叫 `POST /api/content-judgment/start` 且不帶 URL、文字少於 30 字，應回 400 與 `INPUT_INSUFFICIENT_FOR_AUDIT`；帶 URL 或 ≥30 字時應成功，且 system prompt 含 audit workflow、MODE 與評分卡指示。
4. **chat audit**：行為未改動，仍為 workflow=audit、輸入充足度、MODE、Output Schema。
5. **側欄與審判頁**：設定內連結為「角色視角 Overlay」；審判頁說明為「角色視角 Overlay 該視角已發布補充」。
