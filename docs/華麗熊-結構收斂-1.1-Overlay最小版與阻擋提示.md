## 2. 哪些結構化欄位先落地

### Boss 視角（4 欄）

| 欄位 | 型別 | 說明 |
|------|------|------|
| summaryOrder | string[]（頓號分隔輸入） | 摘要優先順序，例：風險、金額、建議動作、結論 |
| summaryLength | enum | 簡短（1–3 行）/ 中等 / 完整 |
| showRiskFirst | boolean | 是否先顯示風險 |
| suggestionPosition | enum | 第一段 / 與結論並列 / 獨立區塊 |

### 投手視角（4 欄）

| 欄位 | 型別 | 說明 |
|------|------|------|
| defaultExpand | string[]（頓號分隔） | 預設展開區塊，例：rescue、scale_up |
| sortPreference | string[]（頓號分隔） | 排序偏好，例：ROAS、CVR、花費 |
| priorityLevel | enum | campaign / product / creative |
| whyNotMoreStyle | enum | 簡短一句 / 獨立段 / 與建議動作合併 |

### 創意視角（3 欄）

| 欄位 | 型別 | 說明 |
|------|------|------|
| lookAtFirst | string[]（頓號分隔） | 先看維度，例：鉤子、前3秒、首圖 |
| outputForm | enum | 先給 3 方向 / 1 完整版 / 並行 |
| outputStyle | enum | 創作（直接給文案腳本）/ 改稿建議（先點問題再給改法） |

儲存時結構化欄位與自由補充一併寫入 DB；組裝時由 `assembleOverlayText` 產出段落再與自由文字合併為單一 Overlay 字串供 prompt 使用。

## 3. Overlay 阻擋提示怎麼改

- **改動前**（偏系統說明）：  
  「此區僅能填寫『視角補充』與『輸出偏向』，不可重寫人格或校準層。偵測到不當內容：xxx。請移除人格級描述，改寫為該視角下的補充偏好、呈現優先順序或輸出規則即可。」

- **改動後**（總監口吻）：  
  **「這裡是視角補充區，不是人格重寫喔。你剛寫的像在改我的角色設定（偵測到：xxx），請改成這個視角下你想先看到什麼、怎麼排優先順序就好。」**

- **位置**：`server/prompt-overlay-validation.ts` 內 `validateOverlayContent` 回傳的 `reason`；`server/routes.ts` 兩處 400 回應的 fallback `message` 一併改為同一句。

## 4. Staging 驗收方式

1. **Overlay 阻擋提示**  
   - 進入「角色視角 Overlay」→ 任選視角 → 草稿輸入「你的最高任務是讓產品更會賣」→ 儲存或發布。  
   - 預期：400，畫面錯誤訊息為「這裡是視角補充區，不是人格重寫喔。你剛寫的像在改我的角色設定（偵測到：最高任務），請改成這個視角下你想先看到什麼、怎麼排優先順序就好。」

2. **結構化欄位**  
   - Boss：填摘要優先順序「風險、金額」、摘要長度「中等」、先顯示風險「是」、建議動作出現位置「第一段」→ 儲存草稿 → 發布。  
   - 預期：儲存與發布成功；已發布摘要（前 3 行）應含上述結構化產出的句子與自由補充（若有）。  
   - 投手 / 創意：各填 1–2 個結構化欄位後儲存、發布，確認無錯誤且摘要可見對應內容。

3. **自動腳本**  
   - `npx tsx script/verify-overlay-validation.ts`：6 case 應全數通過（邏輯未改，僅文案改為總監口吻）。

4. **組裝結果**  
   - 審判頁選該視角並觸發一次審判，確認 system prompt 內 Overlay 段落含結構化產出（若已發布）。
