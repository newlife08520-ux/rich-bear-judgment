# Phase C-2 完成回報

## 1. 這一階段到底解決了什麼痛點

- **影片比例不明時不再誤導**：影片上傳後無法自動抓比例，改為明確顯示「比例待確認」、表單提示「影片比例待確認，請手動選擇」，toast 區分「已帶入類型為影片；比例請手動選擇（待確認）」。
- **大量素材不卡頓**：版本列表改為虛擬捲動（只渲染可見列），單包 80+ 版本時仍可順暢捲動、篩選與多選。
- **素材包數量可感知**：左側素材包列表上方顯示「共 N 個素材包」，方便在包很多時掌握規模。
- **批次流程預留**：多選時出現「批次移動」「批次標記」按鈕（點擊 toast「即將推出」），不擋現有批次刪除，又預留後續擴充。
- **圖片縮圖省流量**：版本卡片內圖片加上 `loading="lazy"`，大量卡片時減少初始請求。

## 4. 目前使用者操作流程變成什麼

1. **登入 → 素材中心**：左側見「共 N 個素材包」、可捲動選包。
2. **選素材包**：右側出現「素材版本」區；有版本時出現篩選列（日期／類型／比例／關鍵字／排序）、批次列（全選、取消選取、批次刪除、批次移動、批次標記）、虛擬捲動的版本卡片網格。
3. **上傳圖片**：選檔上傳 → 自動帶入 MIME、類型圖片、比例（依尺寸）；toast「已帶入檔名、URL、類型與比例」。
4. **上傳影片**：選檔上傳 → 自動帶入 MIME、類型影片、比例預設 1:1 可手動改；toast「已帶入檔名、URL、類型為影片；比例請手動選擇（待確認）」；表單比例欄下顯示「影片比例待確認，請手動選擇」。
5. **版本卡片**：圖片顯示縮圖（lazy）、影片顯示封面或 Film 圖示；影片顯示「比例待確認」、圖片顯示實際比例；可多選、批次刪除；點「批次移動」「批次標記」會看到「即將推出」toast。
6. **大量版本**：列表在固定高度內虛擬捲動，只渲染可見列，數量多時仍順暢。

## 5. Acceptance steps

1. 上傳一影片檔 → 表單類型為「影片」、toast 含「比例請手動選擇（待確認）」、比例欄下有「影片比例待確認，請手動選擇」。
2. 上傳一圖片檔 → 表單類型為「圖片」、比例自動帶入、toast 為「已帶入檔名、URL、類型與比例」。
3. 版本列表有多筆時 → 影片卡顯示「比例待確認」、圖片卡顯示比例數字。
4. 多選版本 → 出現「批次刪除」「批次移動」「批次標記」；點批次移動/標記 → toast「即將推出」。
5. 左側有至少一個素材包 → 列表上方顯示「共 N 個素材包」。
6. 單包 50+ 版本 → 右側版本區可捲動、不卡頓，DOM 節點數明顯少於總筆數（虛擬化生效）。

## 6. Rollback

- **程式**：還原 `client/src/pages/assets.tsx` 到 Phase C-2 前；執行 `npm uninstall @tanstack/react-virtual`。
- **資料**：未改 schema 與後端，無需資料還原。

## 7. 風險與防呆

- **虛擬化 scroll 容器**：虛擬列表的 scroll 容器為 `max-h-[60vh]`，若右側版面被壓縮或 RTL 導致寬度異常，可能需再微調；目前以 60vh 避免佔滿畫面。
- **虛擬化欄數**：目前固定 3 欄（lg），小螢幕用 grid-cols-1/2，虛擬化仍以「列」為單位（每列 3 格），小螢幕會留空位但功能正確；若未來要依 breakpoint 改欄數，可再抽成變數。
- **批次移動/標記**：僅預留按鈕與 toast，未做實際 API；避免使用者以為已可移動/標記，toast 文案為「即將推出」。
- **影片比例**：儲存仍為既有 schema（例如 1:1），「待確認」僅為顯示與提示，不寫入新欄位，避免後端與既有資料不相容。

## 8. 自我檢查後，最可能還殘留的 5 個問題

1. **影片比例仍無法自動帶入**：需後端或前端解析影片 metadata（或上傳 API 回傳寬高），目前僅手動選擇與「比例待確認」提示。
2. **虛擬化在小螢幕的空白格**：sm 為 2 欄、xs 為 1 欄時，最後一列可能不足 3 格，右側會留空；不影響行為，僅視覺。
3. **批次刪除仍為 N 次請求**：大量勾選時延遲與負載較高，未來若有後端 batch delete API 可改為單次呼叫。
4. **篩選/排序未反映在 URL**：重新整理或分享連結無法還原目前篩選狀態。
5. **「共 N 個素材包」在包很多時**：左側列表仍為單一捲動、無虛擬化，若未來包數破百可再考慮左側也虛擬化。
