### 3. 效能與體感說明

- **單包 80+ 版本**  
  - 版本列表使用 `@tanstack/react-virtual`，只渲染可見列（每列 3 張卡、列高 130px、overscan 3 列），理論上 80 筆約 27 列、DOM 僅約十幾列。  
  - **預期**：捲動與篩選應順暢；若仍卡頓，可能來自圖片 lazy load 或單一 scroll 容器內其它大型內容。  
  - **未在 80+ 筆環境實測**，建議您在本機單包建 80+ 版本實際捲動與篩選一次。

- **左側素材包很多時**  
  - 左側為一般捲動列表（無虛擬化），「共 N 個素材包」僅為計數。  
  - 若包數破百，**有可能**在快速捲動時出現輕微卡頓；目前未實作左側虛擬化。

- **可能仍慢或不好用的點**  
  - 批次刪除為 N 次 DELETE，勾選很多筆時延遲與伺服器負載較高。  
  - 篩選／排序僅前端、無後端分頁，單包若破千筆，初次載入與篩選計算會變重。  
  - 自訂日期區間未驗證「結束 &lt; 開始」等邏輯，若填錯仍會送進篩選。

## 三、自我檢討 5 題

### 1. 這一輪離「高效率素材工廠」還差哪 5 件事？

1. **跨包搜尋與全局視圖**：目前關鍵字只搜「當前選中素材包」的包名／產品名與該包內版本檔名，無法一次搜全部包或跨包篩選。  
2. **影片比例與 metadata**：影片無法自動帶入比例、時長、封面，投手需手動選比例或事後補。  
3. **批次移動／標籤**：僅 UI 占位，無法真正「勾選多筆 → 移到另一包」或「批次打標籤」，整理大量素材效率不足。  
4. **後端篩選與分頁**：篩選／排序全在前端，單包極大量時載入與運算壓力大；缺少後端 query 與分頁。  
5. **命名與主素材辨識**：版本卡片未顯示「所屬主素材／系列」、未提供命名範本或自動命名，大量素材時辨識與管理仍吃力。

### 2. 影片比例仍需手動確認，對投手來說有多麻煩？下一輪準備怎麼補？

- **麻煩程度**：每次上傳影片都要手動選一次比例，若一天 20～30 支影片，重複操作多、也容易選錯；若之後要依比例篩選，錯誤比例會影響篩選結果。  
- **下一輪補法**：  
  - 短期：在「新增／編輯版本」表單中，影片類型時將比例欄預設或明顯標示「待確認」，並在列表卡片維持「比例待確認」避免誤導。  
  - 中長期：後端上傳時解析影片 metadata（或使用 ffprobe／雲端轉碼）回傳寬高或比例；或前端用 `<video>` 讀取 metadata 取得 videoWidth / videoHeight 再推比例（需考慮大檔與跨域）。若後端先支援，優先接後端回傳值。

### 3. 若一天 80 支、7 天 500+ 支素材，現在哪裡最先會出現使用壓力？

1. **單包版本數**：若 80 支都塞同一包，右側雖有虛擬捲動，但一次載入 80 筆 API 回應與前端篩選／排序仍會吃記憶體與計算；若 500+ 筆同包，壓力更明顯。  
2. **批次刪除**：勾選 50～80 筆做批次刪除時，會觸發 50～80 次 DELETE，耗時與失敗處理（例如部分失敗）體驗不佳。  
3. **左側素材包數**：若 7 天建 50～100 個包，左側列表無虛擬化，快速捲動找包時可能略卡。  
4. **篩選與關鍵字**：關鍵字只搜當前包，無法「全專案搜某檔名」；日期／類型／比例篩選僅前端，資料量大時仍是一次算整包。  
5. **辨識與命名**：500+ 版本若檔名或命名不統一，僅靠檔名與縮圖辨識主素材／系列仍不足，需要標籤或主素材分組。

### 4. 版本卡片現在最可能還有哪些資訊不夠？

- **時長**：影片沒有顯示秒數（schema 有 `durationSeconds`，但上傳／列表未接），投手無法一眼看出長版／短版。  
- **檔案大小**：未顯示 `fileSizeBytes`，不利判斷是否過大或需壓縮。  
- **封面品質**：影片若無 `thumbnailUrl` 只顯示圖示，無法預覽實際畫面；若有 thumbnail 但未做縮圖規格，大圖可能影響效能。  
- **主素材辨識**：多尺寸同一支創意時，無法標示「同一主素材的不同比例版本」，不利分組與批次建立廣告。  
- **儲存位置**：`storageProvider` 未在卡片顯示，除錯或排查 NAS／本機時較不直覺（可維持不顯眼）。

### 5. 這一輪哪些只是先求可用，不是最佳實作？

- **批次刪除**：用 N 次 DELETE 而非後端 batch API，延遲與錯誤處理都不理想。  
- **日期範圍**：用前端 useMemo 算今天／昨天／7 天／30 天，未與 URL 或後端對齊；自訂區間未驗證起迄合理性。  
- **虛擬化**：欄數固定 3，小螢幕仍以 3 欄為單位虛擬化，最後一列可能留空；未依 breakpoint 動態改欄數。  
- **篩選／排序**：未寫入 URL，重新整理或分享連結無法還原狀態。  
- **影片比例**：用「待確認」提示 + 手動選，而非真正從檔案或後端取得比例。

## 四、是否建議直接進 Phase D

### 建議：**可進 Phase D，但先在本機補完 C-2 畫面與操作驗證**

**理由簡述：**

1. **範圍與方向**：C-2 已守住「只動素材中心、不動左側主架構與分析區」，且圖片／影片型態與比例提示、預覽卡片、篩選／排序／多選／批次刪除、虛擬捲動、左側數量都已到位，與你先前確認的方向一致。  
2. **已完成 vs 占位**：已完成項與「僅占位」已分開標示，不會把批次移動／標記當成已做完。  
3. **站穩的定義**：要「真的站穩」需要你在本機有 1～2 筆圖片／影片版本後，跑過一輪：上傳→預覽→篩選→排序→多選→批次刪除，並在單包 80+ 版本時感受虛擬捲動與左側多包捲動；目前自動化環境無版本資料，無法代你完成這一步。  
4. **Phase D 與 C-2 的關係**：Phase D 是投放中心（CTA、帳號選取、版本區預覽、單尺寸可投、狀態簡化），不依賴 C-2 的「批次移動／標記」或「影片比例自動」；只要素材中心「選版本、看預覽、有篩選」可用，就能接投放流程。  

因此建議：**你先在本機照「一、1」補齊有版本時的畫面證據與操作實測，確認無誤後再開 Phase D**；若你希望更保守，也可先補完 C-2 驗收再進 Phase D。
