# 四階段自動化矩陣建稿 — 架構設計與實作計畫

## 一、階段一：素材中心 — Regex 檔名「自動歸組 (Auto-Grouping)」

### 目標
上傳／解析檔名時，依 `_A(`、`_A_`、`_B(`、`_B_`、`_C(`、`_C_` 等規則，自動對應到「A版」「B版」「C版」主素材組，並寫入 `AssetVersion.groupId`、必要時自動建立 `AssetGroup`。

### 架構設計

| 項目 | 說明 |
|------|------|
| **解析層** | 擴充 `shared/parse-asset-name.ts`（或同目錄新檔），新增 `parseVariantCodeFromFilename(fileName: string): string \| null`，專責從檔名抓 A/B/C 代號。 |
| **Regex 規則** | 優先匹配 `_A(`、`_A_`、`_B(`、`_B_`、`_C(`、`_C_`；可放寬為 `_([A-Za-z])[\s]*[(_]` 取第一組字母為 variant code，再正規化為「A版」「B版」。 |
| **歸組時機** | 上傳完成（`POST .../versions/upload`）回傳 detection 後，前端建立版本時可帶入「建議 groupId」；**或** 在後端 `POST .../versions` 的 service 內，依 `fileName` 呼叫 parser，若解析到 variant 則自動 find-or-create AssetGroup 並寫入 `groupId`。建議：**後端在 create version 時一併做 auto-grouping**，前端只顯示建議、可覆寫。 |
| **AssetGroup 建立** | 若該素材包下尚無名稱「A版」的 AssetGroup，則在歸組時自動建立（`name: "A版"`, `variantCode: "A"`）；若有則直接綁定。 |
| **既有欄位** | `AssetVersion.groupId`、`AssetVersion.groupSource`（suggested/manual）、`AssetGroup.name` 已存在，**不需改 schema**，只需補 parser + 後端 create version 邏輯。 |

### 實作要點
- Parser 與現有 `parseSuggestedGroupNameFromFilename` 可並存：一個回傳「顯示名（A版）」，一個可回傳「代碼（A）」供比對。
- 避免重複建立同名稱 AssetGroup：以 `packageId + name` 查詢現有 group，無則 create。

## 二、階段二：投放中心 — 矩陣式批量建稿 (Matrix Creation)

### 目標
- 受眾/Ad Set 區：可輸入「建立 N 組廣告組合」。
- 素材區：Checkbox 多選「A版」「B版」「C版」。
- 按下「建立草稿」：`(N 個 Ad Set) × (勾選的素材組數) = M 筆草稿`，批次寫入 DB。

### 1. 目前 Schema 是否需擴充？

**結論：現有 `PublishDraft` 與儲存結構可不擴充，即可支援矩陣建稿。**

| 現況 | 矩陣需求 | 做法 |
|------|----------|------|
| 一筆 `PublishDraft` = 一組設定 + 一個 `assetPackageId` + 一組 `selectedVersionIds` | 一鍵產生多筆草稿 | 每筆草稿仍是「一個 draft 對應一個素材組」：矩陣 = 多筆獨立的 `PublishDraft`，每筆的 `selectedVersionIds` = 該組內所有 version 的 id。 |
| `selectedVersionIds` 已為陣列 | 每筆 draft 對應單一主素材組的多個版本（4:5+9:16） | 不變，仍用 `selectedVersionIds` 存該組的 version ids。 |
| 無「Ad Set 模板」實體 | N 個 Ad Set 的差異（如受眾、預算） | 第一版可簡化：N 僅代表「複製同一組設定 N 份」，每份再乘上勾選的素材組，即 `N × 素材組數` 筆 draft；若未來要「每 Ad Set 不同受眾」，再考慮 Ad Set 模板或擴充 draft。 |

因此：
- **不需**新增「矩陣」或「批次」專用表；矩陣結果 = 多筆 `PublishDraft`。
- **不需**改 `PublishDraft` 欄位；可選：若希望區分「同一批矩陣」的草稿，可加 `batchId?: string`（同一批共用一 UUID），屬選配。

### 2. 矩陣邏輯（前端 + 後端）

- **前端**：  
  - 輸入：Ad Set 數量 `N`、勾選的 `groupId[]`（或 group 對應的 versionIds 清單）。  
  - 計算：`drafts = generateDraftMatrix(N, selectedGroupIds, baseSettings)` → 得到 `PublishDraft[]`（每筆含 `assetPackageId`, `selectedVersionIds`, `campaignName`, `adSetName`, `adName` 等）。  
- **後端**：  
  - 維持現有 `POST /api/publish/drafts` 單筆建立；前端對矩陣中每一筆呼叫一次（或新增 `POST /api/publish/drafts/batch` 收 `drafts: PublishDraftCreate[]`，內部迴圈呼叫既有 create）。  

若採 batch API，request body 可為：

```ts
interface PublishDraftBatchCreateRequest {
  drafts: Array<Omit<PublishDraft, "id" | "userId" | "createdAt" | "updatedAt">>;
}
```

## 三、階段三：SOP 自動命名引擎 (Auto-Naming Engine)

以下為**示範用** TypeScript 介面與假資料，確認命名邏輯與欄位對應無誤後再接到表單與矩陣。

### 3.1 命名規則對照（依你提供的 SOP）

| 類型 | 格式範例 | 對應欄位／邏輯 |
|------|----------|----------------|
| Campaign | `轉換次數(原始)0827-品名-3影K+抓住文-T` | 目標前綴 + MMDD + 產品名 + 文案摘要 + 後綴 |
| Ad Set | 可與 Campaign 同或簡化，例如同 campaign 下多 Ad Set 時加序號 | `campaignName` 或 `campaignName + "_" + 序號` |
| Ad（廣告） | 混比例：`(原)混A1+A2+○○文`；單比例：`A1+○○文` | 依該組是否含 4:5+9:16 決定前綴；素材組代號 + 文案前幾字 |

### 3.2 介面與假函數（示範）

```ts
// ========== 階段三：SOP 自動命名引擎（示範介面與 Mock） ==========

/** 命名引擎輸入：來自表單 + 系統日期 + 素材包／主素材組 */
export interface AutoNamingInput {
  /** 目標前綴，如 "轉換次數(原始)" */
  objectivePrefix: string;
  /** 產品名，如 "小淨靈" */
  productName: string;
  /** 文案標題／主文案前幾字，如 "抓住文" */
  headlineSnippet: string;
  /** 系統當天 MMDD */
  dateMMDD: string;
  /** 後綴，如 "T"（可選） */
  suffix?: string;
  /** 該組主素材組顯示名，如 "A版" */
  groupDisplayName: string;
  /** 該組內比例，用於判斷是否「混比例」 */
  aspectRatiosInGroup: Array<"9:16" | "4:5" | "1:1" | "16:9">;
}

/** 命名引擎輸出 */
export interface AutoNamingResult {
  campaignName: string;
  adSetName: string;
  adName: string;
}

/**
 * 動態生成 Campaign / Ad Set / Ad 名稱（依貴司 SOP）。
 * 此為示範實作，實際可再抽成可配置範本。
 */
export function generateSOPNames(input: AutoNamingInput): AutoNamingResult {
  const { objectivePrefix, productName, headlineSnippet, dateMMDD, suffix, groupDisplayName, aspectRatiosInGroup } = input;

  // Campaign：轉換次數(原始)0304-小淨靈-3影K+抓住文-T
  const campaignName = [
    objectivePrefix,
    dateMMDD,
    "-",
    productName,
    "-3影K+",
    headlineSnippet,
    suffix ? `-${suffix}` : "",
  ]
    .filter(Boolean)
    .join("");

  // Ad Set 第一版可與 Campaign 同，或加序號
  const adSetName = campaignName;

  // Ad：若該組同時含 4:5 與 9:16 → (原)混[組名]+[文案前5字]；否則 [組名]+[文案前5字]
  const isMixedRatio =
    aspectRatiosInGroup.includes("4:5") || aspectRatiosInGroup.includes("1:1")
      ? aspectRatiosInGroup.includes("9:16")
      : false;
  const adPrefix = isMixedRatio ? "(原)混" : "";
  const adName = `${adPrefix}${groupDisplayName}+${headlineSnippet.slice(0, 5)}`;

  return { campaignName, adSetName, adName };
}

// ---------- Mock 資料示範 ----------
const mockNamingInput: AutoNamingInput = {
  objectivePrefix: "轉換次數(原始)",
  productName: "小淨靈",
  headlineSnippet: "抓住文",
  dateMMDD: "0304",
  suffix: "T",
  groupDisplayName: "A版",
  aspectRatiosInGroup: ["4:5", "9:16"],
};

const mockNamingResult = generateSOPNames(mockNamingInput);
// 預期約：campaignName = "轉換次數(原始)0304-小淨靈-3影K+抓住文-T"
//         adName = "(原)混A版+抓住文"
```

若「3影K」等為固定字串或來自設定，可再從 `input` 加欄位或常數即可。

## 四、階段四：Meta API PAC / asset_feed_spec 對應 (TypeScript 介面與假函數)

### 4.1 版位對應關係（依你指定）

| 我們的比例 | Meta 版位用途 |
|------------|----------------|
| 4:5、1:1 | 對應 **feed**（動態消息）：Facebook feed、Instagram stream 等 |
| 9:16 | 對應 **story / reels**：Facebook story、Instagram story、Reels |

### 4.2 介面與假函數（示範）

```ts
// ========== 階段四：Meta asset_feed_spec 對應（示範介面與 Mock） ==========

import type { AssetAspectRatio } from "@shared/schema";

/** 單筆草稿 + 綁定之主素材組與版本（用於轉成 Meta 結構） */
export interface DraftWithGroupForMeta {
  draft: {
    id: string;
    assetPackageId?: string;
    selectedVersionIds?: string[];
    primaryCopy?: string;
    headline?: string;
    cta?: string;
    landingPageUrl?: string;
    pageId?: string;
    igAccountId?: string;
  };
  /** 該 draft 對應的主素材組內版本，已依比例分好 */
  versionsByRatio: Array<{
    versionId: string;
    aspectRatio: AssetAspectRatio;
    fileUrl: string;
    assetType: "image" | "video";
    thumbnailUrl?: string;
  }>;
}

/** Meta asset_feed_spec 中，單一影像資產（簡化） */
export interface MetaImageAsset {
  hash?: string;
  url?: string;
  adlabels?: Array<{ name: string }>;
}

/** Meta asset_feed_spec 中，單一影片資產（簡化） */
export interface MetaVideoAsset {
  video_id?: string;
  adlabels?: Array<{ name: string }>;
}

/** Meta 版位標籤：用於 asset_customization_rules */
export type MetaPlacementLabel = "feed" | "story_reels";

/** 我們內部的 build 結果（可再轉成 Meta 實際 JSON） */
export interface MetaAssetFeedSpecBuild {
  /** 依版位分組的影像：4:5、1:1 → feed；9:16 → story_reels */
  images: {
    feed: MetaImageAsset[];
    story_reels: MetaImageAsset[];
  };
  videos: {
    feed: MetaVideoAsset[];
    story_reels: MetaVideoAsset[];
  };
  bodies: Array<{ text: string }>;
  titles: Array<{ text: string }>;
  link_urls: Array<{ website_url: string; display_url?: string }>;
  call_to_action_types: string[];
  /** 對應 Meta 的 asset_customization_rules（示意） */
  placementLabels: Array<{ placement: MetaPlacementLabel; image_label?: string; video_label?: string }>;
}

const FEED_RATIOS: AssetAspectRatio[] = ["4:5", "1:1", "16:9"];
const STORY_REELS_RATIOS: AssetAspectRatio[] = ["9:16"];

/**
 * 將單筆 Draft + 其綁定之主素材組版本，轉成符合 Meta asset_feed_spec 的結構（假函數）。
 * 實際呼叫 Meta 時需再轉成 Graph API 要求的 JSON（含 hash/video_id 上傳結果）。
 */
export function buildMetaAssetFeedSpec(input: DraftWithGroupForMeta): MetaAssetFeedSpecBuild {
  const { draft, versionsByRatio } = input;

  const imagesFeed: MetaImageAsset[] = [];
  const imagesStoryReels: MetaImageAsset[] = [];
  const videosFeed: MetaVideoAsset[] = [];
  const videosStoryReels: MetaVideoAsset[] = [];

  for (const v of versionsByRatio) {
    const label = FEED_RATIOS.includes(v.aspectRatio) ? "feed" : "story_reels";
    const asset = v.assetType === "image"
      ? { url: v.fileUrl, adlabels: [{ name: label }] }
      : { video_id: "[UPLOAD_VIDEO_ID]", adlabels: [{ name: label }] };

    if (v.assetType === "image") {
      if (label === "feed") imagesFeed.push(asset as MetaImageAsset);
      else imagesStoryReels.push(asset as MetaImageAsset);
    } else {
      if (label === "feed") videosFeed.push(asset as MetaVideoAsset);
      else videosStoryReels.push(asset as MetaVideoAsset);
    }
  }

  return {
    images: { feed: imagesFeed, story_reels: imagesStoryReels },
    videos: { feed: videosFeed, story_reels: videosStoryReels },
    bodies: draft.primaryCopy ? [{ text: draft.primaryCopy }] : [],
    titles: draft.headline ? [{ text: draft.headline }] : [],
    link_urls: draft.landingPageUrl
      ? [{ website_url: draft.landingPageUrl, display_url: draft.landingPageUrl }]
      : [],
    call_to_action_types: draft.cta ? [mapCtaToMeta(draft.cta)] : ["LINK"],
    placementLabels: [
      { placement: "feed", image_label: "feed", video_label: "feed" },
      { placement: "story_reels", image_label: "story_reels", video_label: "story_reels" },
    ],
  };
}

function mapCtaToMeta(cta: string): string {
  const map: Record<string, string> = {
    "來去逛逛": "SHOP_NOW",
    "了解更多": "LEARN_MORE",
    "立即購買": "SHOP_NOW",
    "下單": "SHOP_NOW",
  };
  return map[cta] ?? "LEARN_MORE";
}

// ---------- Mock 資料示範 ----------
const mockDraftWithGroup: DraftWithGroupForMeta = {
  draft: {
    id: "draft-1",
    assetPackageId: "pkg-1",
    selectedVersionIds: ["v-4-5", "v-9-16"],
    primaryCopy: "主文案",
    headline: "標題",
    cta: "來去逛逛",
    landingPageUrl: "https://example.com",
    pageId: "123",
    igAccountId: "456",
  },
  versionsByRatio: [
    { versionId: "v-4-5", aspectRatio: "4:5", fileUrl: "/api/uploads/1/a.jpg", assetType: "image" },
    { versionId: "v-9-16", aspectRatio: "9:16", fileUrl: "/api/uploads/1/b.mp4", assetType: "video" },
  ],
};

const mockMetaSpec = buildMetaAssetFeedSpec(mockDraftWithGroup);
// 預期：images.feed 含 4:5 圖；videos.story_reels 含 9:16 影片；bodies/titles/link_urls/call_to_action 來自 draft。
```

實際串接 Meta 時，需再將 `MetaAssetFeedSpecBuild` 轉成 Graph API 的 `asset_feed_spec` 格式（含 `object_story_spec`、`asset_customization_rules` 等），並將 `fileUrl` 換成先上傳後的 `hash` / `video_id`。

## 六、建議實作順序（確認計畫後）

1. **階段一**：Parser（`parseVariantCodeFromFilename`）+ 後端 create version 時 auto-grouping + 必要時自動建立 AssetGroup。  
2. **階段二**：前端矩陣 UI（N 組 Ad Set + 多選素材組）+ `generateDraftMatrix` + 單筆或 batch 建立 API。  
3. **階段三**：將 `generateSOPNames` 接到表單與矩陣產出的每筆 draft，寫入 `campaignName` / `adSetName` / `adName`。  
4. **階段四**：將 `buildMetaAssetFeedSpec` 接到「單筆草稿 → 準備送 Meta」流程，並補實際上傳素材換 hash/video_id 的步驟。

若你確認上述邏輯與命名／版位對應無誤，再依此動工 UI 與 API；若有要調整的命名格式或版位規則，可指定後我再改計畫與示範程式碼。
