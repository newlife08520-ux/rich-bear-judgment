# 設定頁儲存 → 驗證狀態 → 同步串接：重現與修復回報

## 一、問題根因

- **實測結果**：以腳本對本機 `http://localhost:5000` 依序執行「登入 → PUT /api/settings → POST /api/accounts/sync」時，**PUT 與 sync 均回傳 200**，後端在「有效 session + 合法 body」下行為正常。
- **推論**：您端「儲存也失敗」較可能為 **(1) 401 未登入**（session 未建立或未帶 cookie）、**(2) 400 驗證失敗**（例如舊 persisted 資料導致 GET 回傳 enum 為 undefined，表單送出後 schema 失敗）、**(3) 網路/環境**（CORS、連線失敗）。
- **本輪已做**：後端 400 時回傳並記錄詳細錯誤、前端儲存失敗時顯示實際 `err.message`、`getSettings`/載入舊檔時補齊 enum、前端送出前以 `normalizeSettingsPayload` 補齊 enum，避免因 undefined 導致 400。

---

## 二、實際修改檔案

| 檔案 | 變更摘要 |
|------|----------|
| `server/routes.ts` | PUT /api/settings：驗證失敗時 `res.status(400).json({ message, errors: result.error.flatten() })` 並 `console.error` 記錄。 |
| `server/storage.ts` | `getSettings`：無 existing 時回傳完整預設；有 existing 時合併 `severity/outputLength/brandTone/analysisBias` 預設，避免回傳 undefined。`loadPersistedData`：載入舊設定時補齊上述四個 enum 再寫入 store。 |
| `client/src/pages/settings.tsx` | 新增 `normalizeSettingsPayload`，送出前將 enum 補齊為合法值；`saveMutation.mutationFn` 先 `normalizeSettingsPayload(data)` 再 PUT；`onError` 改為顯示 `err?.message`。 |
| `client/src/pages/dashboard.tsx` | 移除重複的 `Dialog` 等 import，修正 Vite 編譯錯誤，使 dev server 可正常啟動。 |
| `script/repro-put-settings.mjs` | 新增：登入 → PUT /api/settings → POST /api/accounts/sync，列印 status 與 body，供重現與驗證。 |

## 三、PUT /api/settings 原本失敗的原因是什麼

- 在本機以腳本重現時 **PUT 未失敗**（status 200，body 為完整 settings 物件）。
- 若您端出現失敗，可依 **Toast 上的錯誤訊息** 判斷：
  - **401**：`未登入` → 請先登入或確認 cookie/session 有帶上。
  - **400**：會帶 `errors`，並在 server console 看到 `[PUT /api/settings] validation failed` 與明細。
- 為避免「舊 persisted 或 GET 回傳 enum 為 undefined → 表單送出 → schema 報錯」，已做：後端 `getSettings`/載入時補 enum、前端 `normalizeSettingsPayload` 再送出。

---

## 四、驗證狀態與 save 的衝突怎麼修

- **邏輯確認**：`valueFingerprint` 對字串做 trim，test-connection 與 saveSettings 皆用同一函數；比對的是「目前值指紋」與「已儲存的 ValidatedValueHash」，只有當「欄位值真的變了」才 clear 驗證，未改動此邏輯。
- **未額外改**：未修改 `patchVerificationStatus` 或 `saveSettings` 的 clear 條件。
- **避免誤判**：`getSettings` 與載入時補齊 enum，避免 refetch 拿到 undefined 導致表單或 fingerprint 比對異常。

---

## 五、同步鏈路怎麼確認真的串上

- 以同一 session 依序執行：
  1. `POST /api/auth/login` → 200，取得 set-cookie。
  2. `PUT /api/settings` → 200，body 為完整 settings。
  3. `POST /api/accounts/sync` → 200，body 為 `{ results: [], message: "未設定任何平台連線，請先在設定中綁定 Facebook 或 GA4" }`（因測試 payload 未填 fb/ga4，屬預期）。
- **結論**：儲存成功後，sync 會讀取 `storage.getSettings(userId)`（剛寫入的設定）；有填 fbAccessToken 或 ga4PropertyId 時，sync 會依現有邏輯呼叫 Meta/GA4 並寫入 syncedAccounts。同步鏈路在「已登入 + 已儲存設定」前提下可正常運作。

## 六、實際重現與修復後的驗收結果

- **重現方式**：`npm run dev` 啟動後，執行 `node script/repro-put-settings.mjs`。
- **結果**：
  - POST /api/auth/login：**200**，set-cookie present。
  - PUT /api/settings：**200**，body 為完整 UserSettings（含驗證欄位）。
  - POST /api/accounts/sync：**200**，results 為 []、message 為未設定平台（因未填 token/property）。
- **修復後**：前端儲存失敗會顯示實際錯誤內容；後端 400 會回傳並記錄 errors；GET/載入與送出前皆補齊 enum，降低 400 與驗證誤清機率。
