## 4. 後端防呆規則怎麼做

- **時機**：POST `/api/workbench/prompts/:mode/draft` 儲存前、POST `/api/workbench/prompts/:mode/publish` 發布前（先 getDraftPrompt 取內容再驗證）。
- **邏輯**：`validateOverlayContent(content)` 用正則偵測人格級語句；任一命中則 `ok: false`，回傳 `reason`、`matchedLabel`。
- **回應**：400 + `errorCode: "OVERLAY_PERSONA_BLOCKED"` + `message: reason`（前端顯示為錯誤文案）。

## 6. 單次 Content Judgment API 如何改成 audit 一致

- **路徑**：POST `/api/content-judgment/start`
- **變更**：
  1. **輸入充足度**：與 chat audit 相同邏輯 — 有 URL 或 text 長度 ≥ 30 才放行；否則 400 + `errorCode: "INPUT_INSUFFICIENT_FOR_AUDIT"` + 說明需提供 URL 或至少 30 字。
  2. **workflow 固定為 audit**：`getAssembledSystemPrompt({ ..., workflow: "audit" })`，不再依預設 clarify。
  3. **組裝結果**：與 chat 的 audit 相同 — 會疊 MODE（依 judgmentType / uiMode）、會加 STRUCTURED_JUDGMENT_OUTPUT_INSTRUCTION。
- **日誌**：console.log 增加 `workflow=audit`。

## 7. uiMode / 視角 / audit mode profile 如何拆責任

- **視角（view role）**  
  - 對應：Boss / 投手 / 創意（uiMode = boss | buyer | creative）。  
  - 責任：選「哪一份」角色視角補充 Overlay（getPublishedPrompt(uiMode)）；不負責人格定義、不負責公式。  
  - 在組裝中：`customMainPrompt = getPublishedPrompt(uiMode)`。

- **audit mode profile**  
  - 僅在 **workflow === "audit"** 時使用。  
  - 責任：決定要疊哪些審判骨架 MODE（A/B/C/D），對應 `UI_MODE_TO_INTERNAL[uiMode]` 或 `judgmentTypeToInternalModes(judgmentType)`。  
  - 在組裝中：`layer3Parts` 在 isAudit 時追加 getModePrompt(m)。

- **程式與註解**：`rich-bear-prompt-assembly.ts` 內 AssembleOptions 註解標明「視角（view role）」與「審判類型；audit 時與 uiMode 一起決定疊哪些 MODE（audit mode profile）」；`UI_MODE_TO_INTERNAL` 註解為「audit mode profile：僅在 workflow=audit 時使用」。

## 8. 哪些東西明確保留在數據引擎，不再進 prompt

以下保留在 **scoring-engine / analysis-engine / routes 計算**，只以 **Data Context 字串**（buildDataContextSection 產出）進 prompt，不把公式或規則寫進人格／Overlay／Calibration：

- 8:2 法則的計算與建議
- 投手公式（預算、擴量、rescue 等）的計算
- Scale Readiness、趨勢與漏斗判斷
- 預算建議、suggestedAction、suggestedPct、reason、whyNotMore
- 成本比、保本 ROAS、目標 ROAS、1d/3d/7d ROAS、TrendCore、Momentum、ATC、花費、營收等

人格真源、Hidden Calibration、角色視角 Overlay 皆不包含上述公式或計算邏輯。

## 9. 驗收方式

1. **Prompt 設定頁**：進入「角色視角 Overlay」頁，確認標題與說明均為視角補充、不出現「主 prompt／人格」編輯語意；側欄連結為「角色視角 Overlay」。
2. **防呆**：在 Overlay 草稿輸入「你的最高任務是…」或「分數哲學」等，儲存或發布應得 400，前端顯示錯誤訊息。
3. **單次審判**：呼叫 POST `/api/content-judgment/start` 且無 URL、text 少於 30 字 → 應回 400、INPUT_INSUFFICIENT_FOR_AUDIT；有 URL 或 ≥30 字 → 應成功且 system prompt 含 audit workflow、MODE 與評分卡指示。
4. **chat audit**：維持不變，輸入不足時仍回 needMoreInput。
5. **數據邊界**：確認未在 Core / Calibration / Overlay 中新增 8:2 或投手公式本體。
